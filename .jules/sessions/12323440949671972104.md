# Architecture, Patterns, and Decisions Summary: Jules Autopilot

## 1. High-Level Architecture
Jules Autopilot is an **Omni-Workspace monorepo** designed as a centralized command center for an ecosystem of independent submodules (e.g., `itgmania`, `borg`, `aios`). It follows a Go-first backend architecture for performance and reliability, with a modern React SPA frontend.

### Stack Breakdown
- **Backend**: Go (Fiber) providing a REST API, WebSocket server, background task queue, and cron scheduler.
- **Database**: SQLite with GORM. Uses WAL mode and connection pooling (`MaxOpenConns: 1`) to handle concurrency safely.
- **Frontend**: React 19 + Vite + TailwindCSS v4. Pure SPA architecture with no SSR overhead.
- **Intelligence**: Multi-provider LLM integration (OpenRouter, Google Gemini, OpenAI) for autonomous operations.
- **Infrastructure**: PNPM Workspaces managing `apps/cli`, `packages/shared`, and the root UI.

## 2. Core Operational Patterns
- **Autonomous Loops**: The system relies on several continuous background processes:
    - **Daemon**: Monitors active Jules sessions, enqueues "nudges" for stalled tasks, and auto-approves low-risk plans.
    - **Shadow Pilot**: Scans for vulnerabilities (dependency audits) and monitors Git diffs to detect regressions.
    - **CI Monitor**: Detects pipeline failures and enqueues autonomous "Auto-Fix" sessions.
    - **Scheduler**: Handles periodic maintenance like codebase re-indexing and log rotation.
- **Task Queue**: goroutine-based semaphore pool handles asynchronous jobs (memory sync, issue evaluation, session spawning).
- **Dual-Layer RAG**: Semantic search across both current source code (`CodeChunk`) and historical session successes (`MemoryChunk`).

## 3. Key Technical Decisions
- **Go as Primary Runtime**: The project pivoted from Bun/Hono to Go to ensure deterministic performance and lower memory footprint for high-concurrency agent orchestration.
- **Single Source of Truth for Versioning**: The `VERSION` file in the root is the master string. All workspace packages (`package.json`, `apps/cli`, `packages/shared`) must be synchronized to this version during builds.
- **Strict Lockfile Hygiene**: Due to CI "frozen lockfile" requirements, any version bump or `package.json` change requires an exhaustive `pnpm install` and a commit of the resulting `pnpm-lock.yaml`.
- **Component Design**: Heavy use of `useSWR` for real-time data fetching with manual revalidation triggers (e.g., Submodule Dashboard, Shadow Pilot Panel).
